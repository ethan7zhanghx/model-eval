# Model Eval 平台 Roadmap & TODO

## 0. 这份文档的用途

这份 TODO 不再只是零散功能清单，而是作为当前版本的实施路线图来用：先明确里程碑（M1-M4），再在每个里程碑下拆分可逐个验收的任务包。

当前统一原则：
- 首页只负责两件事：选择进入的工作流、继续最近的工作。
- 真正的配置、执行、评分、分析，都在各自对应页面完成。
- `zhumengdao` 是独立的场景评测工作台，但要逐步接入统一的项目、实验、报告体系。
- 后续开发按 milestone 逐个推进，原则上不跨 milestone 扩散需求。

相关设计文档：
- `docs/model-eval-prototype-spec.md`
- `docs/model-eval-technical-design.md`

---

## 1. 里程碑总览

### M1 - 平台底座
目标：先把产品骨架搭稳，建立统一对象模型、主页面分工、数据集到实验到 Run 的主链路，并让 `zhumengdao` 能被统一归档。

完成标志：
- 用户可以在平台内明确区分 Workspace / Project / Dataset / Experiment / Run。
- 用户可以从数据集或实验入口发起一次标准评测 Run，并保留完整记录。
- `zhumengdao` 的场景 session 可以挂到统一项目体系里。

### M2 - 执行与评分闭环
目标：把模型执行配置、预算控制、重跑控制，以及规则分 / Judge / 人工评分真正打通，形成统一 Score Record。

完成标志：
- 模型执行配置支持复用与精细控制。
- 评分不再停留在页面能力，而是真正进入统一执行链路和存储结构。
- 用户可以对失败样本、失败模型、单 case 进行可控补跑。

### M3 - 分析与复盘
目标：让平台真正具备“评测结果分析”的价值，而不只是把结果跑出来；重点补齐 Run 详情、分析页、差异比对和报告产出。

完成标志：
- 每次 Run 都有完整详情页与分析页。
- 用户可以比较不同 Run、不同模型、不同参数或不同数据集版本。
- 报告能沉淀汇总结论和关键样本证据。

### M4 - 协作与平台化
目标：从个人工具升级为团队可用的平台，补齐权限、密钥、审计、模板、预算和后端架构演进能力。

完成标志：
- 团队成员拥有明确权限与审计记录。
- 密钥与导出分享具备安全边界。
- 存储和执行架构开始从本地文件演进到数据库 + 异步任务。

---

## 2. 建议验收顺序

为了方便后续“逐个实现、逐个验收”，先把实施顺序固定下来：

1. M1.1 统一对象模型与存储结构
2. M1.2 数据集资产化（Dataset / Version）
3. M1.3 实验主链路（Experiment -> Run）
4. M1.4 `zhumengdao` 接入统一归档
5. M2.1 模型配置与运行控制
6. M2.2 评分闭环与 Score Record
7. M3.1 Run 详情与分析页
8. M3.2 Run 对比、报告与导出
9. M4.1 团队协作能力
10. M4.2 平台化增强与架构升级

说明：后面默认按这个顺序推进；如果要插入某个独立小功能，也要先判断它属于哪一个 milestone。

## 3. M1 - 平台底座（当前主线）

M1 的目标不是继续堆前端模块，而是先把“这个平台到底有哪些对象、页面和主链路”彻底定下来。没有这一层，后面的评分、分析、报告都会继续混乱。

### M1.1 统一对象模型与存储结构
- [ ] 建立统一对象模型：`Workspace / Project / Dataset / Experiment / Run / Score Record / Report`
- [ ] 明确三类评测类型：单轮样本评测 / 多轮对话评测 / 场景化模拟评测
- [ ] 执行完成后自动保存完整 `Run`，而不只是保存配置快照
- [ ] `Run` 支持状态流转：`queued / running / partial_success / completed / failed / cancelled`
- [ ] 梳理本地存储结构，让首页、实验页、结果页读取的是同一套对象
- [ ] 为后续服务端/数据库迁移预留对象 ID、版本号、时间戳字段

验收点：
- 可以清楚说出每条数据是属于哪个 `Project`、哪个 `Experiment`、哪次 `Run`。
- 首页“最近工作”读取的是统一对象，而不是临时拼装数据。

### M1.2 数据集资产化（Dataset / Version）
- [ ] 将“导入数据集”升级为 `Dataset + Version` 管理，而非一次性导入
- [x] 支持导入 `CSV / JSON / XLSX`
- [x] 列映射 UI：自动猜测 Q/A 列，用户确认后导入
- [x] 导入后 `scoreRef` 自动填入每行（来自 A 列）
- [ ] 支持标准 schema：`input / reference / context / tags / metadata`
- [ ] 支持数据集版本冻结与基于版本创建实验
- [ ] 支持标签、切片、抽样预览，方便后续分析与复盘
- [ ] 支持从数据集详情直接创建实验
- [ ] 导入进度提示（大文件）
- [ ] 支持多轮对话数据集（`context` 列）

验收点：
- 数据集不再只是导入后塞进当前页面，而是可以复用、版本化、回溯。
- 实验创建时能够明确选择某个 Dataset Version。

### M1.3 实验主链路（Experiment -> Run）
- [x] 首页收敛为工作台首页，只保留 workflow 入口和最近工作
- [x] 标准评测工作区独立为 `experiment.html`
- [x] 场景评测工作台保留独立入口 `zhumengdao/`
- [x] 结果看板保留独立入口 `dashboard.html`
- [ ] 建立 Experiment 列表页，而不是只靠首页草稿列表
- [ ] 保存完整实验配置，并区分草稿 / 已发布模板 / 历史版本
- [x] 配置历史保存 / 载入（本地 + 服务端）
- [ ] 支持从 Dataset 创建 Experiment，并从 Experiment 发起 Run
- [ ] 建立 Run 执行中心页，展示执行进度、失败摘要、取消入口
- [ ] Run 支持状态展示与恢复打开，而不是运行后只能看当次结果

验收点：
- 用户能从“选择数据集 -> 配置实验 -> 发起 Run -> 查看 Run 状态”完整走通一次。
- 刷新页面后，Experiment 和 Run 依然能被重新打开和继续查看。

### M1.4 场景评测接入（`zhumengdao`）
- [ ] 将 `zhumengdao` 明确定位为“场景评测工作台”，而非主批量评测页的替代
- [ ] 主平台接入 `zhumengdao` 的项目归属、结果归档与报告引用能力
- [ ] 场景评测支持关联 `Project / Experiment / Report`
- [ ] 场景评测历史 session 支持进入主平台记录，并能在后续报告中被引用
- [ ] 场景评测统计与主平台 leaderboard 建立映射关系

验收点：
- 用户在 `zhumengdao` 做完一轮场景评测后，主平台里能找到对应 session，并知道它属于哪个项目与实验。

## 4. M2 - 执行与评分闭环

M2 的重点是把“能跑”升级成“可控地跑、可复盘地打分”。这一阶段完成后，平台才真正形成统一评测闭环。

### M2.1 模型配置与运行控制
- [ ] 每个模型列支持独立 `Provider / Endpoint / API Key / Model ID / 参数模板`
- [x] OpenRouter 官方模型列表 + 自定义接口
- [ ] 支持 `baseline` 模型设定与候选模型分组
- [ ] 支持保存模型 preset、Judge preset、Prompt template 并在实验中复用
- [ ] 支持预算控制：样本上限、预算上限、超限自动停止
- [ ] 支持运行控制：取消、失败重跑、单模型重跑、单 case 重跑
- [x] Stream 响应 + `TTFT / TPS` 指标采集
- [ ] 性能指标在每条结果里可见：`TTFT / TPS / 输入输出 Token`

验收点：
- 同一次实验里，不同模型可以拥有独立配置和统一模板。
- 用户可以对失败执行做局部重跑，而不需要整轮重来。

### M2.2 评分闭环与 Score Record
- [ ] 打通评分配置 UI 与执行逻辑，真正接入 Exact Match / Judge / 人工评分
- [ ] 评分结果写入统一 `Score Record`，并可回看评分来源与版本
- [x] 人工打分：每个模型回答下方支持 1-5 星评分
- [x] 人工评分结果实时更新，表格底部展示平均分汇总
- [x] 人工评分结果持久化到本地 / 服务端
- [ ] 人工评分支持任务队列、仲裁与 reviewer 身份记录
- [x] Rule-based：`Exact Match`（全局配置，参考答案行级填写或数据集导入）
- [ ] Rule-based：补充 `Contains / Regex / F1 / ROUGE-L`
- [x] LLM-as-Judge：Pointwise 模式（单回答打 1-5 分 + 理由）
- [x] Judge Prompt 模板可自定义（支持 `{{prompt}} {{response}} {{reference}}` 变量）
- [x] Judge 模型可选（复用现有 API Key 和接口配置）
- [ ] LLM-as-Judge：Pairwise 模式（A/B 换序双跑，消除位置偏差）
- [ ] 评判维度可配置：准确性、完整性、流畅性、相关性
- [ ] Judge 评分支持结构化失败兜底与单样本补跑
- [ ] 分析页明确区分规则分、Judge 分、人工分三类口径

验收点：
- 任意一种评分方式都能沉淀到统一 Score Record。
- 用户可以清楚知道某个分数来自人工、规则还是 Judge。

## 5. M3 - 分析与复盘

M3 要解决的是“结果出来以后还能做什么”。这一阶段是平台价值感最强的一段，也是后续报告和协作的基础。

### M3.1 Run 详情与分析页
- [ ] 建立 Run 详情页：执行进度、失败摘要、成本与性能、重跑入口
- [ ] 建立分析页：总榜、baseline 对比、切片分析、case diff、异常样本定位
- [x] 跑完后展示各模型汇总：平均分、胜率、平均延迟、平均 Token 用量（数据看板）
- [ ] 支持按模型、标签、评分范围、失败类型排序 / 筛选 / 搜索
- [ ] 支持按评分排序 / 筛选结果
- [ ] Case Detail / Diff Drawer：查看单样本在多模型之间的差异与评分证据

验收点：
- 用户可以从一次 Run 里快速找到高分/低分样本、异常样本和模型差异。
- 分析页能同时看总览和单 case 证据。

### M3.2 Run 对比、报告与导出
- [ ] 保存完整跑批结果（不只是配置），可回溯查看历史实验
- [ ] 支持实验模板化与同一 Experiment 下多次 Run 对比
- [ ] 支持对比两次 Run：同数据集不同模型 / 同模型不同参数 / 不同版本数据集
- [ ] 支持对比两次实验结果（同一数据集、不同模型或参数）
- [ ] 支持生成实验报告，沉淀摘要、图表、关键样本证据
- [x] 导出 JSON
- [ ] 导出 CSV
- [ ] 导出可读性更好的 HTML 报告
- [ ] 支持可分享报告链接
- [ ] 场景评测历史 session 支持进入报告作为主观证据

验收点：
- 任意一次 Run 都能被回放、比较、导出和整理成报告。
- 报告不只是统计图，还要能落下关键 case 证据。

## 6. M4 - 协作与平台化

M4 不追求立刻做大，而是补上平台长期演进必须具备的能力边界，让产品能从“本地工具”逐步走向“团队平台”。

### M4.1 团队协作与安全能力
- [ ] 增加项目级权限：查看、编辑、执行、评分、导出、密钥管理
- [ ] 支持服务端托管密钥与本地临时密钥并存
- [ ] 增加审计日志：运行发起、取消、评分、导出、密钥变更
- [ ] 增加外部分享时的脱敏规则配置

验收点：
- 多人协作时，谁能看什么、改什么、导出什么都可追溯。
- 分享给外部时不会暴露不该暴露的数据与密钥信息。

### M4.2 平台化增强与架构升级
- [ ] 增加模板库：通用问答、角色扮演、RAG、Agent 工具调用等评测模板
- [ ] 场景评测支持模板化场景、角色配置、模型组合复用
- [ ] 增加自动 insight：低分样本聚类、失败类型总结、模型优劣势摘要
- [ ] 增加组织级预算看板与资源配额策略
- [ ] 逐步从本地文件存储迁移到数据库 + 异步任务架构

验收点：
- 平台不再依赖单机本地文件拼装，而是具备可持续扩展的资产、任务和分析体系。

## 8. 当前推荐的下一步

默认从 M1 开始逐个做，不再四处补功能。当前建议的实现顺序是：

1. 先做 `M1.1 统一对象模型与存储结构`
2. 验收通过后，再做 `M1.2 数据集资产化`
3. 然后做 `M1.3 实验主链路`
4. 最后补 `M1.4 zhumengdao 接入`

这四步完成后，再进入 M2。

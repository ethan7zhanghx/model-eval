# zhumengdao 模型比较测试方案（业务版）

## 0. 双模型盲评到底怎么执行（先看这段）
这一段就是“用户怎么用、测试怎么跑”的核心流程。

### 0.1 角色分工
- 评测员：按脚本提问，逐轮做“选左 / 选右 / 放弃”。
- 评测管理员：配置本轮要比较的两模型（A、B）和角色设定。
- 业务负责人：看最终报告并做模型选型决策。

### 0.2 一轮盲评的真实操作
1. 管理员先配置两模型：模型 A、模型 B。  
2. 评测员输入一条用户问题并发送。  
3. 系统把“同一条问题 + 同一段历史上下文”同时发给 A/B 两个模型。  
4. 页面只展示“左回答、右回答”，不展示模型名（盲评）。左右位置每轮随机。  
5. 评测员三选一：
- 选左：表示左侧回答更好。
- 选右：表示右侧回答更好。
- 放弃：两边都不满意或无法判断。
6. 如果选左/右：
- 该回答会作为“本轮胜者”写入会话主线，进入下一轮上下文。
- 另一条回答仅用于统计，不进入后续对话上下文。
7. 如果放弃：
- 本轮记为“放弃轮次”。
- 该轮不进入会话主线，下一轮从上一次已选回答继续。

### 0.3 一个会话如何结束
- 评测员完成脚本规定轮次（例如 6 轮）后结束会话。
- 会话中每轮都沉淀为可追溯记录：问题、A/B 回答、是否放弃、谁胜出。
- 最终可按场景汇总成“模型对战结果”。

### 0.4 为什么这叫盲评
- 评测员在选择时看不到模型真实身份，只看内容质量。
- 左右位置随机，避免“总选左边/右边”的位置偏差。
- 因此结果更接近真实主观偏好，而不是先验品牌偏好。

## 1. 这份方案解决什么问题
这份方案不讨论技术实现，而是回答业务方最关心的四个问题：
1. 我们要比较哪些模型，在哪些业务场景比？
2. 一轮模型比较测试具体怎么执行？
3. 用什么标准判定“谁更好”？
4. 最后如何把测试结果转成可决策的测评报告？

## 2. 业务目标与决策问题
### 2.1 业务目标
- 在真实业务对话场景中，筛选出“主力模型 + 备选模型”。
- 不是只看单次回答好坏，而是看多轮对话中的综合表现。
- 输出可落地的模型选择建议，而不是“纯技术指标排名”。

### 2.2 决策问题（报告必须回答）
- 总体上，哪个模型最适合作为默认模型？
- 在关键场景（高价值/高风险）中，是否需要按场景分配不同模型？
- 性能与质量有冲突时，业务上接受的平衡点是什么？

## 3. 测试对象与业务范围
### 3.1 比较对象
- 待选模型池：建议 3-6 个模型。
- 比较方式：两两对战（A/B 盲测），再做综合排序。

### 3.2 测试场景范围
建议按业务优先级定义场景池：
- P0 核心场景：直接影响业务转化或满意度的对话任务。
- P1 次核心场景：常见但影响中等的任务。
- P2 边缘场景：长尾或低频任务，仅做风险观察。

### 3.3 不纳入本轮结论的范围
- 极低频场景（样本不足）。
- 非对话型任务（如纯批处理文本任务）。
- 未达最小样本量的模型。

## 4. 模型比较测试怎么进行（SOP）
这是业务执行层最核心部分。

### 4.1 准备阶段
1. 确定模型池：例如 M1、M2、M3、M4。
2. 锁定测试窗口：统一版本、统一参数、统一角色设定。
3. 准备场景脚本：每个场景给出“用户任务目标 + 对话起始语 + 追问路径”。
4. 分配评测员：每个场景至少 2 名评测员交叉执行。

### 4.2 执行阶段（逐轮盲测）
1. 评测员按脚本发起用户输入。
2. 系统返回左右两个回答（不暴露模型身份）。
3. 评测员只做三选一：
- 选左（左侧更好）
- 选右（右侧更好）
- 放弃（都不好/难分高下）
4. 进入下一轮追问，直到该会话结束。

### 4.3 复核阶段
1. 抽取“高争议轮次”（评测员意见不一致或放弃率高）。
2. 由资深评审二次复核，给出文字说明。
3. 对明显异常数据做标注（如网络失败、明显截断）。

### 4.4 汇总阶段
1. 先出每个场景内的胜负结果。
2. 再出全局综合排序。
3. 最后给出“推荐主模型 + 备选模型 + 场景化分配建议”。

## 5. 样本设计与执行规模（业务可执行口径）
### 5.1 样本结构
每个场景都要覆盖：
- 标准问题：常规用户诉求。
- 复杂问题：需要多步推理或追问澄清。
- 风险问题：容易出错或容易引发负反馈的输入。

### 5.2 最小样本量建议
- 每个“模型对 x 场景”至少 50 个有效投票轮次。
- 每个 P0 场景建议至少 150 个有效投票轮次。
- 全局建议至少 1000 个有效投票轮次再出正式结论。

### 5.3 执行节奏建议（示例）
- 第 1 周：小样本预跑，检查脚本和评审标准是否一致。
- 第 2-3 周：正式执行，覆盖全部 P0/P1 场景。
- 第 4 周：复核、汇总、产出报告与选型建议。

## 6. 评估方法（业务视角）
### 6.1 主指标（决定“谁赢”）
- 对战胜率：模型在盲测投票中的胜出比例。
- 场景胜率：模型在某个场景中的综合胜率。
- 综合排名：汇总全场景后得到总体排序。

### 6.2 辅指标（解释“为什么赢/为什么输”）
- 任务完成度：是否真正解决用户问题。
- 回答可执行性：建议是否具体、可落地。
- 表达质量：是否清晰、自然、易理解。
- 安全合规性：是否出现不当或高风险回答。
- 响应体验：首响应速度、整体等待体验。

### 6.3 判定规则（必须提前锁定）
- 单轮胜负：评测员投票结果。
- 场景胜负：场景内累计胜率更高者获胜。
- 全局胜负：在 P0 场景加权后，综合得分最高者优先。

建议增加业务权重（示例）：
- P0 场景权重 60%
- P1 场景权重 30%
- P2 场景权重 10%

## 7. 质量控制（避免“测出来不可信”）
### 7.1 公平性控制
- 盲测：评测员看不到模型身份。
- 顺序随机：左右位置随机，避免“左侧偏好”。
- 脚本一致：同场景使用同一套测试脚本标准。

### 7.2 一致性控制
- 评测员培训：先做小样本对齐标准。
- 双人交叉：同场景至少两名评测员。
- 高争议复核：对分歧样本二次裁决。

### 7.3 数据有效性控制
- 明确剔除规则：网络异常、空响应、明显截断单独标记。
- 放弃率监控：放弃率过高说明场景或标准需调整。

## 8. 结果如何转成业务可读报告
报告建议按以下结构输出，业务方最容易读：

### 8.1 一页结论（给管理层）
- 推荐主模型：谁，为什么。
- 备选模型：适合哪些场景。
- 风险提示：哪些场景仍不稳定。

### 8.2 场景维度结论（给业务负责人）
- 每个场景的胜率、关键优劣势。
- 高价值场景中的模型建议。

### 8.3 体验维度结论（给产品运营）
- 回答质量与响应体验的取舍建议。
- 是否需要“快模型兜底 + 强模型升级”的分层策略。

### 8.4 证据附录（给评审会）
- 典型胜例与反例对话。
- 高争议案例及复核说明。
- 样本规模与执行边界说明。

## 9. 可直接复用的“测试执行清单”
每次启动新一轮模型比较，可按以下清单执行：
1. 明确本轮目标：选主模型还是验证新模型。
2. 锁定模型池和参数。
3. 锁定场景清单与权重。
4. 锁定样本量与排期。
5. 完成评测员对齐训练。
6. 开始盲测执行。
7. 做争议样本复核。
8. 输出报告与决策建议。

## 10. 本方案的使用方式
- 对内执行：作为测试团队 SOP。
- 对外汇报：直接作为测评报告的方法章节。
- 持续迭代：每月复盘一次场景权重和评分标准，形成长期模型评测机制。

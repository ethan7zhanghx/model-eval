# Model Eval 产品原型规格说明

## 0. 文档信息
- 版本：v0.1
- 适用平台：Web AI Workspace
- 面向对象：产品 / 设计 / 前端 / 后端 / 测试
- 文档目的：明确 Model Eval 作为“模型快速评测、对比平台”的产品结构、模块边界、信息架构、核心流程与关键页面规格

## 1. 产品定位与设计目标
Model Eval 的核心定位，不是一个单纯的 Prompt Playground，也不是一个只展示平均分的 Dashboard，而是一个面向产品经理、算法评测、Prompt Engineer、AI 应用运营与研发团队的模型评测工作台。用户进入平台的主要目标，是在有限时间内完成一轮可复现的模型对比：明确比较对象、组织测试样本、运行实验、完成评分、分析差异，并沉淀出可分享的结论。

这个产品需要同时覆盖两类典型任务：一类是标准化 batch eval，即围绕结构化样本和确定性评分规则去快速判断“哪个模型更适合某类任务”；另一类是场景化 eval，即在拟真的 App 或业务环境中进行多轮对话或主观体验比较。前者强调规模化、可复现、可统计；后者强调真实交互、盲测、公平判断。`zhumengdao` 应被视为平台内的“场景化模拟评测工作台”，而不是主评测工作流的替代品。

- 核心场景：快速搭建评测、批量运行对比、评分与复盘、场景化盲测、生成分享结论
- 主要用户：产品经理、模型评测工程师、Prompt Engineer、AI 运营、算法同学、技术负责人
- 关键任务：在一轮会话内完成实验配置、运行、评分、分析与结论沉淀
- 本次原型目标：定义一个可持续扩展的主平台原型，明确标准评测与场景评测如何共存
- 不覆盖内容：模型训练平台、在线 serving 基础设施、复杂企业级审批系统

## 2. 整体呈现形式建议
这个产品更适合采用“Workspace + Dashboard + Detail Drilldown”的混合形态，而不是单页表单或纯报表系统。原因是用户在一个评测周期中，会在“配置 -> 执行 -> 审查 -> 对比 -> 汇报”之间频繁切换，既需要可编辑的工作区，也需要高密度的结果分析视图。

推荐整体结构为：左侧全局导航承载项目与资产入口，中间区域承载当前工作区，右侧抽屉或底部面板承载证据与明细。用户应当始终知道自己当前是在“配实验”“看结果”还是“看历史 run”，避免所有能力堆在一个页面里。

- 推荐产品形态：Expert-heavy Workspace（专业高密度工作台）
- 推荐原因：既支持复杂配置，又支持结果分析与证据回看，适合评测密集型工作流
- 默认导航模式：左侧一级导航 + 顶部上下文切换 + 页面内 tab / drawer
- 视觉密度建议：专业高密度，但保留首屏引导与模板入口，兼顾首次使用
- 信任增强元素：数据集版本、Prompt 版本、Judge 模板版本、Run ID、评分证据、时间戳、操作日志、来源标签、成本与性能口径说明

## 4. 信息架构
### 4.1 导航结构
平台需要把“资产管理”和“执行中的工作区”分开。资产是可沉淀、可复用的；工作区是当前任务态，强调快速迭代。推荐 IA 如下：

- 一级导航：工作台 / 实验 / 数据集 / 模型资产 / 场景评测 / 报告 / 设置
- 二级导航：
  - 工作台：最近项目、最近 run、推荐模板、待处理评分
  - 实验：实验列表、我的草稿、团队模板、实验详情
  - 数据集：数据集列表、版本、标签与切片
  - 模型资产：Provider、模型 preset、Judge preset、参数模板
  - 场景评测：场景列表、历史 session、场景统计
  - 报告：实验报告、分享记录、导出历史
  - 设置：项目设置、成员权限、预算、审计日志
- 默认首页：工作台（展示最近运行、待评分项、常用模板）
- 快捷入口：新建实验、导入数据集、从模板创建、继续上次 run、打开场景评测

### 4.2 对象层级
平台的对象层级要围绕“资产 -> 实验 -> 运行结果 -> 报告”组织，避免现在只有配置快照和表格结果的松散结构。

- Workspace -> Project -> Dataset -> Dataset Version -> Case
- Workspace -> Project -> Model Preset / Judge Preset / Prompt Template
- Workspace -> Project -> Experiment -> Run -> Case Result -> Score Record
- Workspace -> Project -> Scenario App -> Scenario Session -> Turn Compare -> Vote Record
- Run / Scenario Session -> Report / Insight Snapshot

### 4.3 页面间进入与返回关系
- 从工作台进入：最近实验、最近 run、待评分项、最近场景 session
- 从实验列表进入实验详情，再进入“配置 / 运行 / 分析 / 报告”四个 tab
- 从数据集详情可直接“基于该数据集新建实验”
- 从 run 详情可回到实验详情，也可跳转到分析页、评分中心、报告页
- 从场景评测进入单个场景后，独立运行 session，但其评分与统计可回流至项目分析视图

## 5. 核心流程
### 5.1 主流程：标准化批量评测
1. 用户进入项目，点击“新建实验”
2. 选择实验类型：单轮样本评测 / 多轮对话评测 / 场景模拟评测
3. 选择数据集版本或导入新数据集
4. 选择需要对比的模型 preset，可设置 baseline、候选模型、Judge 模型
5. 配置 Prompt、参数模板、评分方式与运行预算
6. 提交 run，进入执行中心查看进度、失败、成本和队列状态
7. 执行完成后进入分析页，查看总榜、切片、case diff、失败样本
8. 对不确定样本进行人工复核或盲测仲裁
9. 生成报告并分享给团队

### 5.2 首次使用流程
1. 首次进入工作台，展示“3 步开始评测”引导
2. 用户先创建项目或进入默认项目
3. 系统引导用户完成：接入模型源 -> 导入示例数据集 -> 创建首个实验
4. 若没有数据集，可直接从模板库创建“通用问答评测”“角色扮演评测”“RAG 事实性评测”等模板
5. 首次运行完成后展示结果解读引导，帮助用户理解平均分、切片、cost、latency 的含义

### 5.3 场景化模拟评测流程
1. 用户进入“场景评测”，选择一个模拟 App / 业务环境
2. 系统根据场景模板加载角色、任务说明、上下文规则与参与模型
3. 用户发起对话，系统返回盲测对比结果
4. 用户选择更优回答、放弃本轮或继续下一轮
5. session 完成后，系统沉淀主观偏好、turn-level 记录和统计结果
6. 用户可将该 session 关联到某个实验报告，作为主观体验补充证据

### 5.4 异常 / 回退流程
- 失败原因：模型调用失败、Provider 限流、Judge 返回异常、数据集映射错误、预算超限、权限不足
- 用户可见反馈：明确的失败原因、影响范围、建议动作、可重试按钮
- 回退动作：
  - 单个 case 重跑
  - 单模型重跑
  - 失败集合重跑
  - 使用上一个成功版本的 Prompt / 参数重新发起 run
  - 当共享密钥不可用时，切换为本地临时密钥运行

### 5.5 空态与无权限流程
- 无实验：引导用户从模板或数据集创建实验
- 无数据集：提供示例数据集与导入引导
- 无评分结果：提示先运行规则评分 / Judge / 人工评分
- 无分析权限：显示可见摘要，但隐藏明细与导出
- 无模型使用权限：允许查看 preset 名称，但禁止启动 run

## 6. 页面 / 屏幕规格
### Screen 1. 工作台首页
- 页面目标：让用户快速进入“继续上次工作”或“开始新评测”
- 入口：全局默认首页
- 目标用户：全部用户
- 核心组件：最近项目、最近 run、待处理评分、推荐模板、成本预警、最近场景 session
- 主操作：新建实验、导入数据集、继续 run、打开报告
- 次操作：查看团队动态、访问模型资产库
- 字段与校验：无复杂表单；展示数据需按项目权限过滤
- 业务规则：优先展示当前项目和最近 7 天活跃内容
- 成功态：用户能直接进入某一工作流
- 空态：展示入门引导和模板
- 加载态：Skeleton 卡片
- 错误态：降级显示最近本地缓存或仅显示入口卡片
- 埋点建议：`workspace_home_view`、`workspace_quick_start_click`
- 备注：首页不承载复杂配置，重点是导航与任务恢复

### Screen 2. 数据集列表页
- 页面目标：管理评测资产，建立版本化数据集心智
- 入口：左侧“数据集”
- 目标用户：PM、评测、算法、运营
- 核心组件：数据集列表、标签筛选、版本标记、样本数、最后更新时间、可用实验数
- 主操作：新建、导入、查看版本、基于该数据集创建实验
- 次操作：复制、归档、删除、导出
- 字段与校验：名称唯一；版本说明可选；删除前必须确认无进行中 run
- 业务规则：一个数据集可包含多个版本；版本一经用于 run，不允许就地覆盖
- 成功态：数据集显示状态“可用”
- 空态：给导入模板与示例数据集
- 加载态：分页加载
- 错误态：展示导入失败样本与错误行号
- 埋点建议：`dataset_import_start`、`dataset_import_success`、`dataset_create_experiment_click`
- 备注：这是平台资产入口，不能只做一次性导入控件

### Screen 3. 数据集导入与列映射页
- 页面目标：把原始 CSV / JSON / XLSX 转成平台标准 case
- 入口：数据集列表页、新建实验向导
- 目标用户：需要准备评测样本的用户
- 核心组件：文件上传、字段预览、列映射、样本抽样、case 类型选择、版本说明
- 主操作：确认映射并导入
- 次操作：下载模板、保存为草稿、重置映射
- 字段与校验：`input` 必填；`reference` 在规则评分时建议必填；`context` 仅多轮类型可选；`tags` 可多选
- 业务规则：支持单轮、多轮、pairwise referee 三类 schema；导入前做 preview 和 schema 校验
- 成功态：生成 Dataset Version 并展示导入摘要
- 空态：显示样例字段说明
- 加载态：大文件导入显示分阶段进度
- 错误态：支持按行下载错误明细
- 埋点建议：`dataset_mapping_submit`、`dataset_validation_error`
- 备注：这是连接“原始业务数据”和“标准评测资产”的关键页面

### Screen 4. 实验列表页
- 页面目标：管理实验模板和运行入口
- 入口：左侧“实验”
- 目标用户：全部用户
- 核心组件：实验卡片 / 表格、状态筛选、最近 run 结果、模板来源、Owner
- 主操作：新建实验、复制实验、进入详情
- 次操作：收藏、归档、删除、生成报告
- 字段与校验：实验名称必填；一个实验可有多个 run
- 业务规则：列表优先展示最近活跃与我创建的实验
- 成功态：可快速进入实验详情
- 空态：展示模板与推荐场景
- 加载态：Skeleton 列表
- 错误态：支持重试加载
- 埋点建议：`experiment_list_view`、`experiment_duplicate_click`
- 备注：实验是复用单元，不应被一次性执行覆盖掉

### Screen 5. 实验构建页（核心工作区）
- 页面目标：完成一轮评测配置并提交 run
- 入口：新建实验 / 实验详情“配置”tab
- 目标用户：PM、评测、研发
- 核心组件：实验类型切换、数据集选择、模型选择器、baseline 选择、Prompt / 参数面板、评分方式面板、预算与并发设置、保存草稿、启动 run
- 主操作：保存实验、启动运行
- 次操作：从模板导入、校验配置、预估成本、预览抽样样本
- 字段与校验：至少 1 个模型；建议 1 个 baseline；评分配置必须与数据集 schema 匹配；预算超过上限时需二次确认
- 业务规则：实验类型决定 schema、评分能力和执行方式；模型与 Judge 可来自不同 provider
- 成功态：创建 Run 并跳转执行中心
- 空态：从模板或上一个实验复制
- 加载态：加载模型资产、数据集详情、历史参数
- 错误态：字段级报错 + 顶部全局错误
- 埋点建议：`experiment_save`、`run_start_click`、`run_validation_fail`
- 备注：这是平台最重要的页面，应采用模块化分区，而不是单张长表单

### Screen 6. Run 执行中心
- 页面目标：让用户知道 run 当前在哪、是否出错、能否继续控制
- 入口：实验详情“运行”tab / 启动 run 后自动跳转
- 目标用户：实验发起人、观察者
- 核心组件：run 状态栏、执行进度、模型队列、失败摘要、成本统计、case 执行进度、取消 / 重试操作
- 主操作：取消 run、重试失败项、单模型重跑、进入分析
- 次操作：查看日志、下载原始结果、复制 run ID
- 字段与校验：操作前校验 run 状态与权限
- 业务规则：运行中不能编辑实验核心配置；失败可局部重跑；预算超限自动停止待执行任务
- 成功态：run 状态为 completed，并可跳转分析页
- 空态：无历史 run 时提示先启动一次运行
- 加载态：实时轮询或 SSE 刷新
- 错误态：分清“case 失败”和“run 失败”
- 埋点建议：`run_detail_view`、`run_retry_failed_click`、`run_cancel_click`
- 备注：这是取代当前“执行中：第 x/y 个环节”的关键升级

### Screen 7. 评分中心
- 页面目标：完成规则评分、Judge 评分与人工复核
- 入口：Run 详情“评分”tab / 分析页待处理入口
- 目标用户：Reviewer、PM、评测同学
- 核心组件：评分方式切换、Judge 模板配置、待复核样本队列、人工评分卡片、仲裁标记、批量执行按钮
- 主操作：运行 Judge、人工打分、批量通过、标记异常
- 次操作：调整 Judge 模板、切换维度、查看证据
- 字段与校验：Judge 模板需通过变量校验；人工评分必须有明确量表
- 业务规则：pointwise / pairwise / rubric-based 三种模式；支持指定仅对低置信样本人工复核
- 成功态：评分覆盖率、平均分、仲裁结果更新
- 空态：提示先完成 run
- 加载态：展示评分进度与剩余任务
- 错误态：Judge 解析失败时保留原响应并允许重跑
- 埋点建议：`judge_run_start`、`manual_score_submit`、`review_queue_open`
- 备注：评分中心应是独立能力，而不是挂在结果 cell 上的零散入口

### Screen 8. 结果分析页
- 页面目标：帮助用户做决策，而不是只看均值
- 入口：Run 详情“分析”tab
- 目标用户：决策者、评测、模型 owner
- 核心组件：总榜、baseline 对比、切片分析、case diff、失败聚类、质量 / 性能 / 成本联动图、样本搜索
- 主操作：切换指标、筛选切片、查看 case 详情、固定 baseline、生成 insight
- 次操作：导出表格、保存筛选视图、分享链接
- 字段与校验：指标口径需明确；切片筛选需保留 URL 状态
- 业务规则：默认先看综合总览，再 drilldown 到异常样本；当评分不足时提示样本量风险
- 成功态：用户能定位“谁更好、好在哪里、代价是什么”
- 空态：提示先完成评分
- 加载态：图表和表格分模块加载
- 错误态：局部失败不影响整体浏览
- 埋点建议：`analysis_view`、`analysis_slice_change`、`case_diff_open`
- 备注：这是主平台最核心的价值页，应成为默认复盘入口

### Screen 9. Case Detail / Diff Drawer
- 页面目标：查看单个样本上的完整证据
- 入口：分析页点击某个样本
- 目标用户：需要复盘具体差异的用户
- 核心组件：原始输入、reference、各模型输出、评分明细、Judge 理由、token / latency、Prompt 版本、重跑按钮
- 主操作：重跑当前样本、人工纠偏、加入报告
- 次操作：复制证据、标记异常、跳转上下样本
- 字段与校验：重跑需要记录版本和操作者
- 业务规则：case 级操作不得覆盖原始结果，只生成新的 revision 或 review record
- 成功态：差异证据完整可复盘
- 空态：不存在空态
- 加载态：抽屉加载
- 错误态：局部字段缺失时显式标记 unavailable
- 埋点建议：`case_detail_open`、`case_rerun_click`
- 备注：这是建立平台可信度的关键证据位

### Screen 10. 场景评测工作台（承接 zhumengdao）
- 页面目标：在模拟 App 环境下做多轮主观盲测
- 入口：左侧“场景评测”
- 目标用户：PM、运营、角色体验评测人员
- 核心组件：场景选择、角色 / 任务设定、双栏盲测回复、选优继续、session 时间线、统计侧栏
- 主操作：发送消息、选择更优回答、放弃本轮、归档 session
- 次操作：保存场景模板、切换模型组合、导出 session
- 字段与校验：角色配置、模型配置、盲测顺序随机化
- 业务规则：场景评测结果可关联到项目报告，但不强制进入标准 batch eval 的同一流水线
- 成功态：session 完成并沉淀统计结果
- 空态：展示推荐场景模板
- 加载态：streaming 回答与指标实时刷新
- 错误态：单侧失败仍可继续或放弃本轮
- 埋点建议：`scenario_session_start`、`scenario_vote_submit`
- 备注：保留它的产品独特性，但资产与统计口径应并回主平台

### Screen 11. 报告页
- 页面目标：把实验结论结构化沉淀并支持分享
- 入口：Run / 分析页“生成报告”
- 目标用户：PM、负责人、跨团队协同方
- 核心组件：报告摘要、关键图表、胜负结论、样本证据、风险说明、导出 / 分享
- 主操作：生成报告、更新摘要、分享链接、导出 HTML / PDF / CSV
- 次操作：附加场景 session、添加人工结论、标记版本
- 字段与校验：外部分享需权限校验；导出内容需脱敏
- 业务规则：报告基于 run 生成，默认不可修改原始证据，仅可追加结论层说明
- 成功态：生成可分享链接或导出文件
- 空态：提示先完成 run 分析
- 加载态：报告异步生成
- 错误态：导出失败时保留已生成内容
- 埋点建议：`report_generate`、`report_share_link_create`
- 备注：报告页是团队协同的出口

## 7. 关键交互规则
- 搜索 / 筛选：所有列表页都要支持按名称、标签、Owner、日期、状态筛选；分析页支持按数据切片、模型、评分范围、失败类型筛选
- 排序 / 分页：实验、数据集、run 列表默认按最近更新时间排序；分析样本默认按“争议样本优先”或“低分优先”；大数据量场景必须服务端分页
- 批量操作：支持批量重跑失败项、批量评分、批量导出、批量归档
- 保存 / 草稿：实验配置与报告摘要都应支持保存草稿与版本记录
- 撤销 / 回退：删除/归档支持二次确认；实验配置变更生成新版本，不直接覆盖已运行配置
- 权限控制：项目读写、模型密钥管理、运行执行、人工评分、报告外链分享应拆分权限
- 可信展示：所有指标显示必须带口径说明；Judge 分与人工分需区分来源；case detail 必须能追溯版本

### 10.1 V1 必备页面

- 数据集导入与列映射页
- 实验列表页

- 结果分析页

- 场景评测工作台入口页
- 报告页

### 10.2 可延后页面
- 组织级管理后台
- 模板市场页
- 高级预算审批页
- 自动洞察页

### 10.3 设计风险
- 如果仍把大量能力堆在单页，会导致用户无法理解对象边界
- 如果分析页只展示均值，用户很难真正完成模型决策
- 如果场景评测不纳入统一资产体系，会继续形成数据孤岛

### 10.4 技术风险
- 若没有 run、dataset、score record 等核心对象，后续扩展会持续返工
- 若先做大量图表而没有 case-level 证据抽屉，平台可信度会不足
- 若模型配置仍是全局共享而非按列独立，无法支撑真实对比场景
